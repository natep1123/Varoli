# 00 — Varoli G3 Primitives Overview

*Last Updated: March 28, 2026*

**Status:** Working reference  
**Layer:** Ground 3 / Primitive-Building-Blocks  
**Authority:** Build-layer overview  
**Scope:** Defines the role of the primitive inventory, the three primitive classes, and the basic rules governing their use in Varoli generation.

---

## 1. Purpose

This folder contains the immediate generative foundation of Varoli.

Its purpose is to provide a compact, searchable, and learnable reference for:
- the primitive classes
- the primitive inventory
- the invariant function of each primitive
- the core interaction rules that govern valid primitive generation

This is not a conceptual overview and not a higher-order grammar layer.
It is the build-layer reference from which directional forms, stable worlds, and later structural development proceed.

---

## 2. Primitive classes

Varoli has three primitive classes only.

### Substrates
Realized as vowels.

Substrates define ontological domains or conditions.
They do not act as forces.

### Kinetics
Realized as consonants.

Kinetics define qualities of motion.
They do not act as domains.

### Operator
Realized as `y`.

The operator governs superposition and collapse.
It is neither substrate nor kinetic.

---

## 3. Core rules

### 3.1 Non-overlap
The three classes do not overlap.

- a substrate may not behave as a kinetic
- a kinetic may not behave as a substrate
- the operator may not be treated as either class

### 3.2 Invariant function
Each primitive has one invariant function.

Context may affect:
- position
- scope
- scale realization
- compositional role

Context may not change primitive identity.

### 3.3 Derivation over invention
All valid forms must reduce to primitive functions and current interaction rules.

If a proposed form cannot be derived from:
- primitive inventory
- directionality rules
- stable-form rules
- current Ground 3 developments

then it should not be treated as valid structure.

### 3.4 Readability matters
Compressed or extended forms must remain structurally readable.
A form that is elegant but unreadable should not be treated as successful generation.

## 7. Use rule

When generating or validating a new primitive-level form:

1. identify the class of each primitive
2. confirm invariant function
3. confirm directionality
4. confirm whether the result is primitive, derived, stable, layered, or exploratory
5. reject any form that requires hidden mechanics or class overlap
