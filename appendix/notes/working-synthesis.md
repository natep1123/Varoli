# Working Synthesis

## Current Decision

Status is intentionally under-specified for certain underivable forms for now.

Development proceeds through validated patterns rather than forcing a premature formal status architecture.

This is the current finding within the tested architecture, not a claim that the question is permanently closed.

---

## Status Indeterminacy

### What has been established

- Primitive contradiction provides a clearer invalidity bridge.
- Underivable local-compositional forms have been identified and tested.
- Underivability does not mechanically determine ontological status.
- No derivational bridge from underivability to ontological status has been established in the tested local-compositional architecture beyond the primitive-contradiction case.

### What remains mechanically undeterminable

For forms such as:
- `V + CV₁ + CV₂`
- `V + VCV₁ + VCV₂`

current tested architecture does not mechanically determine whether they are:
- invalid
- incomplete
- lawful unresolved
- or another formal status category

### What status claims are justified

- Primitive function violation can justify invalidity more clearly than other tested cases.
- Underivable forms may remain status-underspecified within current tested architecture.
- Mechanical undeterminability is a valid formal result.

### What status claims are not justified

- Underivable = invalid
- Underivable = incomplete
- Underivable = lawful unresolved
- Validated-pattern absence = automatic status consequence

---

## Validated-Pattern Sufficiency

### What validated patterns are available

- Primitive classes remain stable:
  - standalone vowels as perspective markers
  - CV / VC as directional forms
  - VCV as stable centered forms
  - `y` as operator
- `V + CV + VCV` is the clearest validated local resolution pattern.
- CV-mediated multi-stable-center governance is provisionally supported in limited scope.
- Recurrent local resolution behavior is structurally usable even without a completed general status architecture.

### What those patterns support structurally

- Local resolved meaning-units
- Controlled chaining from validated material
- Further development from structurally clear forms
- Bounded exploration of compression from validated sources

### Why development can continue from them

The language already has enough validated material to continue productive development without first solving all status questions for underivable forms.

### What they do not settle

- Full positive validity criteria
- Full status architecture
- General ontological classification of underivable forms
- Full extension of governance beyond validated scope

---

## Revisit Conditions

Status architecture should be revisited if any of the following happens:

- Compression requires explicit distinction between valid, incomplete, and underivable source forms
- Chaining or discourse development fails without stronger status distinctions
- A new untested domain produces a real bridge from derivability to status
- Schema pressure becomes actual evidence rather than speculation
- A broader architectural layer cannot proceed honestly with status left under-specified

Status architecture does **not** need to be revisited yet merely because:
- some forms remain underivable
- bridge-testing failed in tested local domains
- the system lacks a total status theory

### Future results that would reopen the bridge question

- A derivable rule that links underivability to a specific status consequence
- A compositional domain where status distinctions become mechanically necessary
- A new primitive or schema pressure that is evidenced rather than conjectured

---

## Compression Preconditions

### Purpose

Begin the next bounded productive domain using only validated patterns.

### What prior work makes available

- Stable primitive inventory
- Validated local resolution pattern: `V + CV + VCV`
- Limited CV-mediated governance support
- Honest status under-specification where required

### What compression should start from

Compression should start only from structurally validated forms and locally resolved units.

### What compression must not assume

- Compression does not automatically solve status architecture
- Compression does not justify a source structure retroactively
- Underivable forms should not be treated as compression-ready by default

### Smallest forms to test first

- `V + CV + VCV`
- other already-validated CV-mediated local units only if needed

### What counts as success

- identifying the smallest structurally valid compression source
- preserving structural identity through reduction
- staying entirely within validated material

Use validated patterns for forward development.

Do not force ontological status claims for underivable forms unless a new derivational bridge is actually established.
